# Standard ワークフロー（標準プラン）

## 適用条件

- 中程度の複雑さ
- 3-10ファイルに影響
- 既存パターンの活用が可能

## 適用例

- 新機能追加
- 複数ファイルのリファクタリング
- API エンドポイントの追加
- データベーススキーマの変更

## プロセス

### 1. 目的理解

- ユーザーの要求を正確に理解
- スコープと期待される成果を定義
- 前提条件を確認

### 2. 情報収集

- `Grep`/`Glob`でコードベースを検索
- 関連ファイルと既存パターンを把握
- 依存関係を分析
- 技術的制約を確認

**完了条件**:
- [ ] 対象ファイルの特定完了
- [ ] 既存パターンの把握完了
- [ ] 依存関係の理解完了
- [ ] 技術的制約の把握完了

### 3. 設計検討

- `WebFetch`で最新情報を取得
- 業界標準の設計パターンを参照
- ベストプラクティスを確認
- 複数の実装案を比較

### 4. プラン作成

- 実行可能なアクションアイテムに変換
- Phase/タスク単位で整理
- 完了条件を明記
- リスクと対策を含める

### 5. 検証

設計検討後とプラン作成後の2回検証:

#### 設計検討後
- [ ] 設計案が要件を満たしているか
- [ ] ベストプラクティスに準拠しているか
- [ ] リスクが特定されているか

#### プラン作成後
- [ ] 完全性: すべての要件がプランに反映されているか
- [ ] アクション可能性: 各タスクは具体的なアクション動詞で始まるか
- [ ] 測定可能性: 完了条件は客観的に判断可能か
- [ ] 依存関係: タスク間の依存が正しく反映されているか
- [ ] リスク対応: 主要なリスクへの対策が含まれているか

### 6. 出力

- `ai/plans/YYMMDD_[概要].md`に保存
- ステータス: Ready for Implementation
